# Plan 12-01: Wire Protected Product Journeys To Real APIs

<objective>
Connect the remaining frontend screens to real NestJS/Supabase-backed workflows so volunteers, ONGs, and admins can complete the MVP from the UI.
</objective>

<tasks>

## Task 1: Add authenticated frontend API utilities and route guards

**Type:** frontend/security

**Files:**
- `frontend/src/hooks.server.ts`
- `frontend/src/routes/+layout.server.ts`
- new frontend server utility under `frontend/src/lib/`

**Action:**
- Centralize backend fetch with bearer token from Supabase session.
- Load current user/domain profile enough for nav and route decisions.
- Add server-side redirects or blocked states for volunteer-only, ONG-only, admin-only, pending/rejected ONG, and suspended account routes.

**Verify:**
- `npm --prefix frontend run check`
- Manual route attempts for logged-out/protected pages.

**Acceptance Criteria:**
- Protected routes do not render demo content to unauthorized users.

## Task 2: Implement volunteer application journey

**Type:** frontend/backend-integration

**Files:**
- `frontend/src/routes/oportunidades/[id]/+page.svelte`
- `frontend/src/routes/oportunidades/[id]/+page.server.ts`
- `frontend/src/routes/minhas-inscricoes/+page.svelte`
- `frontend/src/routes/minhas-inscricoes/+page.server.ts`

**Action:**
- Add real apply action to opportunity detail.
- Show login/register CTA only when unauthenticated.
- Show deadline/duplicate/application status states.
- Replace static `minhas-inscricoes` with backend data.

**Verify:**
- Volunteer can apply once and see status.
- Duplicate application returns clear message.

**Acceptance Criteria:**
- INT-01 and INT-02 application status path work through UI.

**Acceptance Criteria:**

## Task 4: Implement ONG application review

**Type:** frontend/backend-integration

**Files:**

**Acceptance Criteria:**
- INT-04 works through UI.
