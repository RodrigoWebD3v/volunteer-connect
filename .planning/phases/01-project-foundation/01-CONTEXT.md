# Phase 1: Project Foundation - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Establish a runnable full-stack project foundation with backend NestJS, frontend SvelteKit, TypeScript, linting, formatting, documented local commands, and a smooth local development experience. This phase does not implement domain features, authentication, Supabase models, or database schema beyond any local infrastructure needed to make the workspace runnable.

</domain>

<decisions>
## Implementation Decisions

### Repository Structure

- **D-01:** Use a simple folder structure, not a formal package-workspace monorepo.
- **D-02:** Prefer top-level folders named `backend` and `frontend` so the academic project stays easy to navigate.
- **D-03:** Keep shared packages out of scope for phase 1 unless a generated tool requires minimal root configuration.

### Package Manager And Scripts

- **D-04:** Use `npm` as the package manager.
- **D-05:** Provide clear root-level commands where practical, so a developer can run backend, frontend, and checks without remembering many nested commands.
- **D-06:** Keep backend and frontend runnable independently as well, because the project may be worked on by different team members.

### Quality Baseline

- **D-07:** Cover the setup fully from the beginning: TypeScript checks, ESLint, Prettier, and test scaffolding should be configured in phase 1.
- **D-08:** The first phase should include commands that prove the generated backend and frontend are healthy before committing.
- **D-09:** README documentation should treat quality commands as part of normal development, not optional extras.

### Local Development Experience

- **D-10:** Use Supabase remoto for the database to avoid local PostgreSQL/Docker setup friction.
- **D-11:** Keep local development focused on running backend and frontend with npm.
- **D-12:** The phase should include `.env.example` and documented default ports for backend and frontend.
- **D-13:** Database schema and migration decisions remain deferred to Phase 2, using Supabase connection variables.

### the agent's Discretion

- Exact NestJS and SvelteKit scaffolding commands.
- Exact root script names, as long as they are documented and ergonomic.
- Whether Docker Compose starts only infrastructure in phase 1 or also includes app containers, based on simplicity and reliability.
- Exact test runner defaults generated by NestJS/SvelteKit, as long as checks run successfully.

</decisions>
