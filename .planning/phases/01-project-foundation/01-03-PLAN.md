# Phase 1 Plan 03: Local Workflow And Documentation

<objective>
Create the root developer workflow: npm helper scripts, Docker Compose local infrastructure, environment examples, gitignore coverage, and README instructions that make the backend and frontend easy to install, run, validate, and stop.
</objective>

<context>
Read before execution:

- `.planning/phases/01-project-foundation/01-CONTEXT.md`
- `.planning/phases/01-project-foundation/01-RESEARCH.md`
- `.planning/REQUIREMENTS.md`
- `AGENTS.md`
- `.planning/phases/01-project-foundation/01-01-SUMMARY.md` if it exists
- `.planning/phases/01-project-foundation/01-02-SUMMARY.md` if it exists

This plan depends on backend and frontend scaffolds being present. It may configure local infrastructure services, but it must not create Supabase schema, migrations, domain tables, auth endpoints, or product UI.
</context>

<threat_model>
## Security Threat Model

**Assets:** local environment examples, Docker service configuration, README commands, root scripts.  
**Threats:** real secrets committed through `.env`; exposing database ports without clear development-only framing; root scripts masking failing child checks; Docker service readiness confused with schema readiness.  
**Mitigations:** commit only `.env.example`; add `.env` and local generated files to `.gitignore`; document Docker as local development only; make root scripts fail when any child command fails; explicitly state schema/migrations belong to Phase 2.
</threat_model>

<tasks>
  <task id="T1" type="auto">
    <title>Add root npm workflow scripts</title>
    <files>
      <file>package.json</file>
      <file>package-lock.json</file>
      <file>backend/package.json</file>
      <file>frontend/package.json</file>
    </files>
    <action>
      Create a root `package.json` using npm, without configuring npm workspaces. Add root scripts that delegate with `npm --prefix backend ...` and `npm --prefix frontend ...`: `dev:backend`, `dev:frontend`, `dev`, `lint`, `format`, `check`, `test`, and `verify`. Install any small root dev dependency needed to run backend and frontend concurrently, such as `concurrently`. Keep backend and frontend package scripts independently usable.
    </action>
    <verify>
      npm run lint
      npm run format
      npm run check
      npm test
      npm run verify
    </verify>
    <acceptance_criteria>
      - Root `package.json` exists and uses npm scripts only.
      - Root scripts delegate to `backend` and `frontend`.
      - `npm run verify` runs all phase-level quality checks.
      - Backend and frontend package scripts still run independently.
    </acceptance_criteria>
  </task>

  <task id="T2" type="auto">
    <title>Add Docker Compose and environment examples</title>
    <files>
      <file>compose.yaml</file>
      <file>.env.example</file>
      <file>.gitignore</file>
    </files>
    <action>
      Add `compose.yaml` for local infrastructure. Include a `postgres` service with a named volume, development-safe placeholder credentials sourced from `.env`, and host port `5432`. Add `.env.example` with documented values such as `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `POSTGRES_PORT`, `BACKEND_PORT=3000`, and `FRONTEND_PORT=5173`. Update `.gitignore` so `.env`, local Docker leftovers, `node_modules`, build outputs, and generated coverage folders are not committed. Do not add Supabase schema or migrations.
    </action>
    <verify>
      Test-Path compose.yaml
      Test-Path .env.example
      docker compose --env-file .env.example config
    </verify>
    <acceptance_criteria>
      - `compose.yaml` validates with Docker Compose.
      - `.env.example` contains development placeholders only.
      - `.gitignore` prevents committing real `.env` files and generated dependency/build folders.
      - No database schema or migration files are created.
    </acceptance_criteria>
  </task>
