# Plan 11-01: Stabilize Schema, CI, And Environment

<objective>
Make the project internally consistent and verifiable before deeper product integration: clean Supabase schema path, safe seed, green CI, documented dependency/security posture, and accurate environment docs.
</objective>

<tasks>

## Task 1: Reconcile Supabase migrations

**Type:** database

**Files:**
- `supabase/migrations/0001_modelo_inicial.sql`
- `supabase/migrations/0003_oportunidades.sql`
- `supabase/migrations/0004_inscricoes.sql`
- `supabase/migrations/0005_presencas.sql`
- new corrective migration if needed
- `docs/database/schema-proposto.md`

**Action:**
- Compare all backend `.from(...)`, `.select(...)`, `.insert(...)`, `.update(...)`, and RPC usage against migrations.
- Fix schema drift for `oportunidades`, `inscricoes`, `presencas`, statuses, FK names, and indexes.
- Decide and document one clean path: consolidated reset for academic dev database or idempotent alter migration.
- Ensure status values match code: `reprovada`, not mixed with `rejeitada`, unless both are intentionally migrated.

**Verify:**
- SQL review for column names used by backend services.
- `rg -n "ong_id|voluntario_id|vagas|rejeitada|participacoes|eventos" supabase backend/src docs/database`

**Acceptance Criteria:**
- Clean database path can produce the columns backend expects.
- No contradictory status vocabulary remains in active docs/migrations.

## Task 2: Add safe seed and validation notes

**Type:** database/docs

**Files:**
- `supabase/seed/`
- `docs/database/README.md`
- `docs/delivery/demo-local.md`

**Action:**
- Provide seed data for demo/verification without real CPF/CNPJ or real emails.
- Include users/ONGs/opportunities/applications/presences enough for phases 12-13.
- Document order: migrations, bucket setup, seed, app run.

**Verify:**

- `rg -n "SUPABASE_SERVICE_ROLE_KEY|cpf|cnpj|@gmail|@hotmail" supabase/seed docs/database`

**Acceptance Criteria:**
- Seed supports at least one volunteer, one approved ONG, one pending/rejected ONG, one published opportunity, one application, and one presence.

**Type:** code-quality

- `backend/src/oportunidades/oportunidades.service.ts`
- `backend/src/presencas/presencas.service.ts`

**Acceptance Criteria:**

## Task 4: Resolve dependency audit posture

**Type:** security

**Files:**
- `backend/package.json`
- `backend/package-lock.json`
- `frontend/package.json`
- `frontend/package-lock.json`
- `docs/verification/final-security-review.md`

**Action:**
- Run audit on backend and frontend.
- Apply safe `npm audit fix` where possible.
- If a finding cannot be fixed safely, document package, advisory, impact, and mitigation.

**Verify:**
- `npm --prefix backend audit --audit-level=moderate`
- `npm --prefix frontend audit --audit-level=moderate`
- `npm run ci`

**Acceptance Criteria:**
- No unexplained moderate-or-higher audit finding remains.
